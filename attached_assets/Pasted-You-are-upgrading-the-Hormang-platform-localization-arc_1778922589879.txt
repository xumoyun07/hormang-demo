==================================================
IMPORTANT ARCHITECTURE RULE
===========================

There are TWO translation systems:

---

1. STATIC UI TRANSLATIONS

---

Examples:

* buttons
* menus
* errors
* settings labels
* login text

These MUST use i18n locale files.

Example:
locales/uz/common.json
locales/ru/common.json
locales/en/common.json

Use:

* i18next
  OR existing localization library already installed.

==================================================
2. DYNAMIC CONTENT TRANSLATIONS
===============================

Examples:

* categories
* survey questions
* plans
* FAQ
* announcements
* badges
* onboarding texts
* admin-created content

These MUST NOT use static locale JSON files.

Instead use multilingual objects.

==================================================
REQUIRED TRANSLATION STRUCTURE
==============================

DO NOT USE:

titleUz
titleRu
titleEn

DO NOT USE:
hardcoded strings.

USE THIS STRUCTURE EVERYWHERE:

{
id: "cleaning",

title: {
uz: "Tozalash",
ru: "Уборка",
en: "Cleaning"
}
}

==================================================
CREATE GLOBAL TYPES
===================

Create reusable localization types.

Example:

type LocalizedText = {
uz?: string;
ru?: string;
en?: string;
};

==================================================
CREATE TRANSLATION HELPER
=========================

Create reusable helper:

getLocalizedText(field, currentLanguage)

Behavior:

1. Return current language
2. If missing:
   fallback to Uzbek
3. If Uzbek missing:
   fallback to Russian
4. If Russian missing:
   fallback to English
5. Never crash UI

Example:

getLocalizedText(category.title, "ru")
