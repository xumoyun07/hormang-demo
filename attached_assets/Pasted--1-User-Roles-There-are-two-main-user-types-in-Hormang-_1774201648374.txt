### 1. User Roles

There are two main user types in Hormang:

- Buyer (Client / Customer)  
  - Needs services or products and books/orders them.  
- Provider (Master / Service Seller)  
  - Offers services (and optionally small products) and receives orders.  

Each user must choose a role once at registration; switching later is allowed only through a simple profile setting or admin panel (not automatic).

***

### 2. What Information Is Needed at Registration

#### For Buyers (mandatory + optional)

Mandatory:
- Name  
  - First name (required).  
  - Last name (required).  
- Contact Info (one method, user‑choice)  
  - Email OR phone number (user selects one preferred).  
- Password  
  - At least 8 characters.  
  - Mix of at least:  
    - lowercase,  
    - uppercase,  
    - digit,  
    - special symbol (optional but recommended).  
- Role selection  
  - Explicit choice: “I’m a buyer (customer)”.

Optional (can be added later in profile):
- Profile photo (avatar).  
- Default language (if multiple languages supported).  
- Location / city (for future personalized recommendations and search bias).  
- Notification preferences (push, email, SMS).  

Profile is basic at first, but can be completed later.

***

#### For Providers (Masters / Service Sellers)

Mandatory:
- Name  
  - First name.  
  - Last name (if applicable; can be business name in later extensions).  
- Contact Info  
  - Email OR phone (same as buyer).  
- Password (same rules as buyer).  
- Role selection  
  - “I provide services (provider / master)”.  
- Service categories  
  - At least 1–3 service categories selected from a predefined list (e.g., Cleaning, Repairs, Nanny, Cooking, Delivery, Crafts, Photography, Massage, Auto, etc.).  
- Short description / bio  
  - 2–4 sentences: what they do, experience, who they work with, etc.  
- Working areas  
  - City/region where they work (e.g., “Tashkent, within 10 km” or “Shahrisabz, city only”).

Optional but highly encouraged:
- Profile photo (avatar).  
- Portfolio  
  - 3–10 photos or small media items (before/after, workspace, finished projects).  
- Working hours / availability  
  - Free text or structured (e.g., “Monday–Friday 10:00–20:00”).  
- Service details  
  - Typical price range, common jobs, target customer (families, students, offices, etc.).  
- ID or business verification hints (for future “verified” badge):  
  - Hint: “You can add identification documents later for a verified badge.”  
- Preferred contact method (WhatsApp‑like, SMS, in‑app chat, phone, etc.).  

Provider profile is marketing‑focused from the start; it must be rich enough so buyers can trust and choose.

***

### 3. Login Flow (for Both Buyers and Providers)

#### Login Screen (Same UI, Role‑based Redirect)

- Inputs:
  - Email or phone (same as used at registration).  
  - Password.  
- Buttons:
  - “Log in”  
  - “Forgot password?” (opens password‑recovery flow).  
- Links:
  - “New user? Register” (opens registration screen).  
  - Optional: “Remember me” (stores a secure session token on device).  

#### Backend Login Logic

- Verify email/phone + password.  
- If valid:  
  - Issue JWT access token (short‑lived, e.g., 1 hour).  
  - Issue refresh token (long‑lived, 7–30 days, stored securely).  
  - Store last_login_at in user record.  
- Role‑based redirect:  
  - role = buyer → Buyer Home / Services Feed.  
  - role = provider → Provider Home / Requests / Orders.  

#### Security Rules

- Password hashing (bcrypt / Argon2).  
- Rate‑limiting on login (e.g., 5 failed attempts → 5‑minute lock or OTP challenge).  
- Generic error messages (no “email not found” leak):  
  - “Incorrect email/phone or password.”  
- HTTPS‑only for all auth endpoints.  
- Optional but professional:  
  - OTP‑based verification (SMS/email) at first login or device change.  
  - SSO options (Google, Apple, Telegram, etc.) as alternative to email/phone + password.

### 4. Profile Structure in the Whole Platform

Once logged in, each user has a public profile and a private profile section.

#### 4.1 Public Profile (Visible to Others)

For Buyers (Client):

- Basic info:  
  - Name, profile photo.  
  - City / location (optional, but helpful for trust).  
- Activity signals:  
  - “Order history” hint (without details) can be exposed as “Has completed many orders” (text only, no full list).  
  - Reviews they left (as text, attached to services/products).  
- Trust signals:  
  - If provider reviews a buyer, can show: “Friendly customer”, “Clear communicator”, etc. (text tags, no number‑based ratings).  

For Providers (Masters / Sellers):

- Basic info:  
  - Name, profile photo, location, working radius.  
  - Bio + “about” text.  
- Services offered:  
  - List of service categories.  
  - Main service types (e.g., “Home cleaning”, “Deep cleaning”, “Car interior cleaning”).  
- Portfolio:  
  - Gallery of 3–10+ images.  
- Reputation layer:  
  - Average rating (1–5 stars).  
  - Number of reviews.  
  - Top‑level review quotes (short text snippets).  
- Verifications / badges (future but designed from start):  
  - “Phone verified”,  
  - “Email verified”,  
  - “ID‑verified” (optional, with admin help).  
- Availability / working hours:  
  - Text or structured “Mon–Fri 10:00–20:00, weekends by request”.  

This public profile is shown on every service/product card, request list, and chat screen as a clickable card or minimal badge.

***

#### 4.2 Private Profile (Only Visible to User)

Shared sections for both roles:

- Account details:  
  - Email, phone, password‑change.  
  - Security settings (two‑factor, device management).  
- Notifications:  
  - Push, email, SMS settings.  
  - Order‑related, marketing, or announcement toggles.  
- Language & preferences:  
  - Preferred language, units (e.g., currency, distance).  
- Support / Help:  
  - Contact support, report a problem, FAQ.  

Buyer‑specific:

- Order history:  
  - List of past orders (status, provider, date, price).  
  - Ability to re‑order or request similar service.  
- Saved / Favorites:  
  - Marked providers or products.  
- Payment methods (if saved):  
  - Card (last‑4 digits, type) — optional UI, with clear privacy labeling.  

Provider‑specific:

- Earnings / Wallet view:  
  - Balance (if Hormang uses escrow or partial‑hold system).  
  - Transaction history (withdrawals, credits, bonuses).  
- Leads monetization (if present):  
  - “Credits” balance,  
  - “Subscriptions”,  
  - “Promotions” (paid boosts).  
- Service listings / catalog:  
  - List of all services and products they’ve created.  
  - Bulk edit, pause, hide options.  
- Reviews they received:  
  - Read all reviews, respond to reviews (if allowed).  
- Business information:  
  - Business name, VAT/tax ID (optional, for formal markets).  
  - Links to external sites or social profiles (optional).

### 5. How Profiles Work Across the Whole Platform

#### 5.1 Profile Discovery & Search

- Buyers see provider cards in:  
  - Home feed,  
  - Search results,  
  - AI‑search results,  
  - Request response lists.  
- Each card shows:  
  - Avatar,  
  - Name,  
  - Rating (stars + count),  
  - Short tagline (from bio or service description).  
- Clicking profile → full profile page with gallery, reviews, detailed description, working hours, contact options.

#### 5.2 Profile as a Trust Layer

- Ratings and reviews are always visible near the profile image.  
- Verification badges (email/phone/ID) appear as small icons or labels.  
- Chat screen shows profile snapshot: avatar, name, rating, short “bio” line, working hours, distance (if relevant).  
- Orders include provider snapshot (name, avatar, rating) so buyers can always see who they’re dealing with.

#### 5.3 Profile Updates and Moderation

- Users can edit:  
  - Name, bio, portfolio, working hours, services, contact info, photos.  
- Some fields (email/phone, role, ID‑related) can be locked or require admin verification.  
- Admin tools allow:  
  - Reviewing suspicious profiles,  
  - Removing or blocking profiles,  
  - Flagging incomplete or misleading info.  
- Public profiles can be partially hidden (mark as “under review”) while backend validation is ongoing.

***

### 6. End‑to‑End User Flow: From Registration to Working Profile

1. Registration  
   - Buyer/Provider chooses role, provides basic contact info + password.  
   - Provider adds service categories, short bio, working area.  
2. Login  
   - User logs in with email/phone + password; JWT is issued.  
3. Profile completion  
   - User is guided (optional) to upload photo, add more details (bio, working hours, portfolio).  
4. Using the platform  
   - Buyer browses, sends requests, chats, orders, reviews.  
   - Provider receives requests, sends proposals, fulfills orders, collects reviews and ratings.  
5. Profile evolution  
   - Profile accumulates:  
     - Number of orders,  
     - Reviews,  
     - Rating,  
     - Verified badges,  
     - Portfolio updates.  
6. Admin oversight  
   - Admins can review profiles, flag or block, set “verified” status for quality providers.
